第六章：把 AI 放进游戏开发现场

这个专题把游戏引擎、代码 Agent、美术资产生成和 MCP 工具连接放在同一条链路里。目标不是证明 AI 能一键生成游戏，而是学会把一个小游戏小样做到可运行、可审查、可复盘。

开始第 21 课 看游戏落地目录
游戏落地学习顺序
  1. 先判断 AI 在游戏生产链路里的位置。
  2. 再把引擎任务拆成 Codex 可执行小工单。
  3. 接着让 Agent 写玩法代码，并做审查和试玩。
  4. 然后把 AI 美术放进资产管线。
  5. 最后用 MCP 连接设计、引擎和验收工具。

进入主讲义打开工作纸

用第一性原则看游戏 AI

先拆生产链路，再选模型、Agent 或 MCP。

1. 玩法玩家反复做什么，3 分钟内能看到什么反馈。
2. 引擎哪些对象、组件、脚本和场景要进入 Unity、Unreal 或 Godot。
3. 代码把玩法拆成状态、事件、数据、边界条件和测试。
4. 资产区分概念、占位和正式资产，保留来源、授权、格式和性能记录。
5. 连接用 MCP 把设计、引擎、文件和验收工具接进受控工作流。

游戏开发中 AI 最适合先做什么

先从低风险、高频、可验证的环节开始。

原型

灰盒玩法切片

让 Agent 拆工单、写小脚本、生成测试场景说明，先验证玩法是否成立。

代码

状态机和组件草稿

把敌人 AI、拾取、技能、背包这类功能拆成状态、事件和参数，再交给 Agent 写初版。

美术

概念图和占位资产

用生成图探索方向，正式使用前必须补版权记录、人工修整、导入规格和性能检查。

UI

设计到实现清单

用 Figma MCP 读取设计上下文，再生成 UMG、Unity UI 或前端实现要求。

测试

试玩脚本和问题归纳

让 Agent 整理测试路径、复现步骤、Bug 分类和下一轮修复工单。

治理

权限、日志和回滚

越靠近真实项目，越要限制写权限、保留日志、人工确认和版本回滚。

好任务

给游戏 Agent 的需求怎么写

  • 说明引擎版本、入口场景、脚本目录和可改范围。
  • 把玩法写成状态、事件、参数、边界条件和通过标准。
  • 要求输出运行步骤、试玩步骤、风险清单和回滚方式。
  • 资产生成必须记录来源、提示词、授权状态和导入规格。
风险边界

不要把 AI 当完整游戏团队

  • 不要把 AI 生成图直接当可商用正式资产。
  • 不要让 Agent 无确认地删除、批量重命名或提交项目。
  • 不要只看代码能编译，不做引擎运行和人工试玩。
  • 不要忽略 Steam 等平台对生成式 AI 内容的披露要求。

4 个上手练习

从一个小样开始，建立“玩法 -> 引擎 -> 代码 -> 资产 -> MCP”的闭环。

练习 1：拆一个小游戏核心循环

写出玩家动作、目标、失败条件、奖励反馈和 3 分钟验收方式。

练习 2：写一个引擎工单

限定场景、脚本和对象范围，让 Codex 只做一个可运行变化。

练习 3：做一张资产管线表

为 10 个图标写用途、来源、提示词、授权、尺寸、格式和导入设置。

练习 4：画 MCP 工作流

从 Figma 读取 UI，到测试引擎项目生成原型，再截图对比和人工确认。
