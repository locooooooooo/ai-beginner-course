实战演示 01 · Codex 本地工具迭代

用 Codex 迭代世界杯策略看板

这个案例来自本地目录 D:\世界杯。它不是教人下注，而是演示怎样把一个高风险决策问题，拆成数据、规则、看板、复盘和审计五层，让 Codex 一轮一轮把想法变成可检查的本地工具。

看迭代流程 先看风险边界
本案例学什么
  1. 如何把模糊策略变成文件结构和服务边界。
  2. 如何让 Codex 写生成脚本、Web 看板和文档库。
  3. 如何用复盘和审计规则约束模型幻觉与过度自信。
风险边界

这是一篇工程教程，不是投注建议

原项目涉及赔率、注单、资金和赛果复盘。公开教程只保留工程方法：数据如何分层、规则如何沉淀、看板如何展示、复盘如何反推下一轮规则。具体盘口、金额、下注组合和实时建议不作为教程内容发布。

项目最终长什么样

Codex 不是只写一段分析，而是把策略工作拆成一个本地可运行系统。

数据层

读取实时缓存、人工录入、参考报告、历史账本和策略导出，所有来源都要有时间戳和可信度标签。

规则层

把“能不能买”“该不该买”“买多少”“怎么展示”拆开，写成可复用规则和 JSON 配置。

服务层

用本地 HTTP 服务提供 /api/state、/api/health、/api/refresh 和文档 API，前端只负责渲染。

看板层

展示刷新状态、数据来源、策略排序、赔率变化、组合窗口、文档库和复盘结果。

复盘层

每日记录结果、命中和失败原因，用反事实回放判断新规则能否减少旧错误。

审计层

用脚本检查规则版本、数据源、敏感残留、相关性暴露和误导性措辞，防止越改越危险。

Loop Engineering：这个项目怎么迭代

每一轮都必须产出文件、运行结果或复盘证据。

1
定义目标

先把目标从“给我策略”改成“做一个可刷新、可展示、可复盘的本地策略看板”。

2
拆目录

把项目拆成 data、records、outputs、web、logs 和脚本入口，让每类产物有固定位置。

3
先跑通最小链路

生成策略 JSON、Markdown 和 HTML 预览，再让 Web 页面读取 API 渲染。

4
引入复盘

真实结果出来后，不问“这次为什么没中”这么粗的问题，而是记录哪个规则导致暴露。

5
升级规则

把复盘教训写回 rules，比如普通平不能被让球平替代、串关公共腿不能重复过多。

6
写审计脚本

让 Codex 把规则变成自动检查，避免下一轮又出现同类错误或危险措辞。

目录怎么读

看一个 Codex 项目时，先看边界，不要先钻进最长的代码文件。

入口文件

README.md

说明服务端、Web、API、启动命令、配置方式和当前展示字段，是理解项目的第一入口。

核心生成

build_worldcup_draw_strategy.py

把赛事、赔率、报告和规则整合成逐场策略行，输出 CSV、Markdown 和 JSON。

服务状态

worldcup_service.py

负责刷新状态、保存快照、计算变化、管理记录和复盘信息，是后端状态核心。

本地 API

worldcup_server.py

提供 /api/state、/api/health、/api/refresh、/api/docs，并托管 web/ 静态页面。

规则沉淀

records/rules/

保存迭代规则、模型口径和 JSON 版本，让下一次 Codex 接手时不会只靠聊天记忆。

复盘证据

records/daily-review/

记录每天的失败、命中、误判和下一轮修正，是这个项目真正变强的来源。

从失败里升级规则

这个项目最值得学的不是“预测”，而是失败后如何改系统。

失败不是只改结论

6 月 15 日复盘暴露的是公共腿重复、精确比分过窄、追加资金没有分散风险。下一轮要改仓位和相关性规则，而不是只换一场比赛判断。

命中也不等于模型成功

6 月 16 日接近保本，但复盘指出模型漏掉了同日普通平集群。命中一张票不能掩盖结构性机会被削弱。

反事实回放检验新规则

用新规则回放旧日期，检查是否能减少灾难性亏损。反事实不是证明自己神准，而是验证规则是否真的修了旧漏洞。

新手可以复用的 Codex 提示方式

同类高风险工具，不要让 Codex 直接“给答案”，而是让它先搭系统。

好提示

让 Codex 做工程闭环

  • 先读取目录，输出数据层、规则层、服务层和展示层。
  • 新增功能前先说明会改哪些文件、怎么验收。
  • 每次策略失败后，先写复盘，再把复盘变成规则。
  • 把危险措辞、过期数据、相关性暴露做成审计脚本。
坏提示

不要把 Codex 当算命工具

  • 不要问“今天怎么买稳赚”。
  • 不要让 AI 用未验证数据直接给资金建议。
  • 不要把未加权枚举说成盈利概率。
  • 不要因为一次命中就放大仓位。

验收清单

做完类似工具，至少要能回答这些问题。

文件验收

README 能讲清项目入口；records 能追溯规则；outputs 能复现导出；logs 能排查服务。

运行验收

本地服务能启动，/api/health 返回状态，/api/state 有快照，前端能刷新。

内容验收

所有高风险输出都标注数据来源、更新时间、可信度和适用边界。

复盘验收

每次结果后都有复盘文档，并能说明下一轮具体改了哪条规则。

本案例的第一性原则

AI 不应该替你承担风险，它应该帮你暴露风险

对高风险决策工具来说，Codex 的价值不是让结论更像专家，而是把依据、边界、失败、复盘和审计都显性化。能被检查、能被回放、能被修正规则的系统，才比一次漂亮回答更有价值。
