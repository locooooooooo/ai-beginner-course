# AI 新手课文字系统与内容框架

## 关联规范

本文件负责课程内容、讲法、练习和验收。视觉风格的 Source of Truth 在：

- `design-system/ai-beginner-course/MASTER.md`
- `design-system/ai-beginner-course/pages/`

后续迭代时先判断改动属于哪一类：

- 改课程内容、课时结构、工作纸字段：先看本文件。
- 改颜色、字体、卡片、按钮、贴纸、移动端布局：先看设计系统。
- 改生成逻辑：改 `scripts/generate_lessons.py`，然后重新生成 `lessons/`、`site/`、`github-pages-publish/`。

## 课程定位

这套课面向零基础学习者，不把 AI 当成名词百科讲，而是把每一课做成可跟做的教学单元。学习者每打开一节课，都应该知道：

1. 这节课解决什么真实困惑。
2. 先记住哪一句人话。
3. 哪 3 个概念最重要。
4. 可以照着哪个场景练。
5. 最后要交付什么小产物。

课程不追求一节课只塞进一个页面。每节课至少有两个层次：

- 主讲义页：负责讲清概念、关系、案例、误区和方法。
- 练习工作纸页：负责让学习者写下输入、过程、检查点和最终产物。

每节课还要保留一组“评价闭环”：

- 学习元信息：预计用时、难度、需要工具、隐私边界、更新时间、来源提示。
- 深度讲稿：人话解释、老师讲法、跟做案例、完整示范。
- 参考答案：字段名为 `reference_answer`，不是唯一答案，而是判断方向是否正确的参考思路。
- 小测验：先让学习者自己答，再展开参考思路。
- 评分标准：至少区分通过、优秀、需要重做。
- 术语表：把本课关键词翻译成新手能复述的人话。

## 文字风格

### 总体口吻

- 用产品经理和 AI 工程师共同授课的口吻：先讲业务目标，再讲技术结构，最后讲落地验收。
- 面向新手，不堆术语。每个术语都要回答“它帮我解决什么问题”。
- 句子短，先给结论，再解释原因。
- 多用“你现在要做什么”“怎么判断做对了”“下一步怎么升级”。
- 少用抽象形容词，多用可执行动作。

### 固定表达顺序

每个知识点都按这个顺序写：

1. 先说人话：一句话解释，不超过 120 字。
2. 再拆结构：输入、处理、输出、检查。
3. 放进场景：办公、电商、内容、客服、知识库或工程项目。
4. 给出动作：让学习者马上写、改、查或画一张流程。
5. 留下证据：要求一个可检查的小产物。

### 禁止风格

- 不把 ChatGPT、DeepSeek、Dify、Codex 等产品名当成知识本身。
- 不把“会拖节点”“会写提示词”包装成完整能力。
- 不给没有验收方式的练习。
- 不把高风险输出说成可以直接相信。
- 不用纯概念堆叠替代真实案例。

## 内容主题框架

### 第一段：正确认知

目标是把 AI、模型、产品、Agent、MCP、Skill 和边界拆清楚。学习者先知道这些词分别站在哪一层，不急着深入实现。

核心产物：一张自己的 AI 能力分类表。

### 第二段：基础使用

目标是让学习者能问出好问题、写基础提示词、拆复杂任务、检查结果。这里是所有后续工作流的基础。

核心产物：一套自己的提问模板和验证清单。

### 第三段：日常场景

目标是把 AI 放进办公、学习、内容、图片、视频、知识库问答里，先看到成果，再理解限制。

核心产物：一个可复用的日常任务模板。

### 第四段：工作流与系统

目标是从单次问答升级到可重复流程，理解 Agent、MCP、Skill 如何组合成系统。

核心产物：一条带输入、工具、输出、验收的工作流图。

### 第五段：电商落地

目标是把前面的能力放到内容生产、获客、私域、客服、复购和全局系统里，形成业务闭环。

核心产物：一张电商 AI 落地路线图。

### 第六段：游戏场景落地

目标是让学习者看到 AI 如何进入真实游戏开发：游戏引擎原型、Agent 编写玩法代码、美术资产生成、Figma/Unity/Unreal 等 MCP 工具连接、运行验收和风险治理。

核心产物：一份 AI 辅助游戏开发小样方案，包含玩法目标、引擎任务、代码 Agent 分工、美术资产规范、MCP 工具连接和验收清单。

### 老鸟进阶：Codex 与工程化

目标是让已经完成入门训练的学习者看到 AI 不只回答问题，还能进入真实项目做读取、修改、运行、验证、发布和复盘。

核心产物：一个能交给 Codex 执行的任务说明，以及一条从本地修改到线上验证的工程化流程。

## 每节课的标准结构

### 主讲义页

1. 核心问题：为什么这节课值得学。
2. 新手速读：一句话先抓住本质。
3. 学习路线：4 步跟做。
4. 关键概念：只抓 3 件事。
5. 第一性拆解：从目标、结构、边界拆开。
6. 最小原理：迁移到其他场景仍成立的判断。
7. 真实案例：至少 2 个可读案例。
8. 应用方法：拿到工作里怎么用。
9. 反常识提醒：最容易犯的错误。
10. 完成标准：本课小产物和验收方式。
11. 参考答案：给出可对照的判断思路，不替学习者编唯一标准答案。
12. 小测验和评分标准：让学习者知道自己是否真的掌握。
13. 术语表：把高频词保留成可复习的小词典。

### 练习工作纸页

1. 课前定位：我为什么要学这节课。
2. 概念翻译：把术语翻译成人话。
3. 场景拆解：选择一个场景，写输入、AI 动作、人工检查、输出。
4. 跟做任务：照本课练习生成产物。
5. 风险检查：列出不能直接相信的部分。
6. 复盘升级：写出下一次要补的上下文、工具或数据。
7. 小测验：先填写自己的答案，再展开参考思路。
8. 对照改进：用参考答案和评分标准修改产物。
9. 术语复述：把本课术语改写成自己的表达。

工作纸里的填写区使用可输入的 `textarea`。页面不做自动保存，必须提醒学习者复制到自己的笔记或文档。

## 游戏场景落地表达口径

讲游戏开发里的 AI、Agent、Codex、UEMCP、Unity MCP、Figma MCP 时，要始终强调：

- AI 不是一键生成完整游戏，而是加速原型、代码草稿、资产探索、测试、文档和复盘。
- 游戏是可体验系统，最终判断要靠运行、试玩、性能、版权、平台披露和人工审美。
- 引擎任务要拆成小工单：场景、对象、组件、脚本、输入、运行方式和验收证据。
- Agent 写玩法代码前要先写清状态、事件、数据、边界条件、测试方法和禁止改动范围。
- AI 美术优先用于概念、占位和变体探索，正式资产必须走来源记录、授权复查、人工修整、导入规格和性能检查。
- MCP 是连接工具的协议，不是无限授权；真实项目要保留只读优先、人工确认、日志、版本控制和回滚机制。

## Dify + Codex 表达口径

讲 Dify + Codex 时，要始终强调：

- 项目应该尽量开源、可本地部署、可审计。
- Dify 是流程编排平台，Codex 是能进入项目现场执行和验证的工程助手。
- 价值不在一次演示，而在可复用工作流资产：workflow、prompt、checklist、template、sample、acceptance card、rule library。
- 工作流资产要进入 Git 管理，包括 Dify 项目、Docker 配置、网关项目、工作流模板和工具流配置。
- 目标是形成“观察真实执行 -> 提炼问题和成功模式 -> 结构化成资产 -> 注入工具链 -> 验证 -> 迭代”的循环。

## 验收标准

一节课算完成，不看页面数量，只看证据：

- 学习者能用自己的话解释这一课。
- 学习者能把概念套进至少一个真实场景。
- 学习者完成了一个小产物。
- 页面给出了检查标准，而不只是内容展示。
- 后续课程能复用这节课的产物继续推进。
