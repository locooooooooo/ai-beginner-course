一句话案例

Codex 不只是写文案，而是把内容推进到可交付

这个项目从一个故事想法开始，最后形成一套可以继续生产第 02 集的内容工程。核心价值不是某一段文字写得好，而是把“小说”和“视频”中间的所有转换步骤都显式化、文件化、可复用。

项目定位

先把题材、冲突和结尾钩子固定下来，后面的文件才不会跑散。

题材

古装轻喜剧、穿越、客栈群像、微悬疑。

剧情核心

现代短视频编导林舟穿进同福客栈，靠“客栈生存簿”救场并留下。

结尾钩子

半页账纸上出现“葵花派”，为第 02 集埋下悬疑线。

Codex 实际做了这 8 步

这条链路的重点是把内容创作变成可复用工程。

1
确定题材和钩子

先明确穿越、客栈群像、轻喜剧和微悬疑，不让内容跑散。

2
生成小说正文

先写完整故事，保证人物关系、冲突和结尾统一。

3
压缩视频脚本

把正文改成适合旁白、口播和短视频节奏的脚本。

4
拆镜头分镜

按画面、动作、对白和情绪拆成可生成的视频镜头。

5
整理结构化 JSON

让每个场景都有时长、画面提示词、动作、旁白和字幕。

6
补字幕和角色 bible

用 SRT 和角色一致性设定控制节奏与人物外观。

7
组合 HTML 成片工程

用本地 HyperFrames 工程承载 15 个场景，减少外部依赖。

8
导出 review 成片

先生成 review MP4，再根据效果决定是否出高质量 final。

可迁移经验

真正值得复用的是链路，不是某一段文案。

先有完整故事

先写小说正文，再拆视频，故事内核更稳。

先定角色一致性

角色外观和性格设定要尽早写，不然画面生成容易前后变化。

用结构化数据交接

JSON 比纯文本更适合进入视频流水线，也更容易被检查和复用。

先 review 再 final

先导出低成本 review 成片，再决定是否投入高质量制作。

复用模板

下一次可以直接套这个顺序

题材设定 -> 小说正文 -> 视频脚本 -> 分镜文案 -> JSON 数据 -> 字幕 -> 角色 bible -> HTML 成片 -> MP4 review。
