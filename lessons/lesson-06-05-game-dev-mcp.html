为什么要学

核心问题

为什么 MCP 对游戏开发重要？因为游戏项目里最难的不是让 AI 回答，而是让它安全地看见项目、调用工具、修改场景、读取设计并留下证据。

人话解释

MCP 可以理解成 AI 和工具之间的连接协议。游戏开发里，它能让 Agent 读取 Figma 设计、查询引擎场景、调用脚本或整理文件，但每个工具能读什么、能写什么、何时需要人确认，都必须提前设边界。

先把 MCP 讲成连接层

不要把 MCP 讲成神秘模型。它解决的是 AI 怎么安全地访问外部工具、数据和动作。游戏项目工具很多，所以 MCP 特别有用。

再分三类游戏场景

Figma MCP 适合设计到 UI 实现，Unity MCP 或 UEMCP 适合引擎和编辑器自动化，文件系统和脚本 MCP 适合资源整理、日志和检查。

顺手把智能体职责讲清

这节课不要只讲 MCP 工具，还要顺手讲清不同智能体的职责。Codex 更适合工程交付；OpenClaw 更像常驻个人助理；Hermes Agent 更强调长期记忆、自改进和持续跟进。

最后强调权限

MCP 越强，越要小心。读取设计稿和列文件风险较低，批量改场景、删除资源、提交发布必须有人确认和可回滚。

跟做案例

从 Figma 血条设计到引擎 UI 原型

  1. Figma MCP：读取血条、技能按钮、背包格子的尺寸、颜色、间距和状态。
  2. Agent：生成 UI 实现清单和组件命名，不直接改正式项目。
  3. 引擎 MCP：在测试场景创建 UI 预制体或 UMG 草稿。
  4. 验收：截图对比设计稿、记录差异、人工确认后再进入正式分支。

输出样例

工作流应包含：Figma 节点读取、样式抽取、UMG 组件清单、测试项目生成、截图对比、人工审批和回滚点。风险包括设计 token 缺失、字体不可用、控件状态不完整、引擎版本差异和权限过大。

检查点
  • 是否先只读，再在测试项目写入。
  • 是否有人工确认点和禁止事项。
  • 是否有截图对比、日志和回滚。
